## 不是工具的错，是 workflow 的问题

很多人用了一周 AI 编程工具就放弃了，理由是"不够准"、"经常乱改代码"、"不知道它做了什么"。

问题不在 AI，在于**没建立正确的工作流**。

## 核心原则：把它当实习生，不是搜索引擎

AI 的优势是**执行**，不是判断。

```
搜索引擎：你知道问题是什么，你知道答案在哪
AI：你知道目标是什么，它负责执行
```

正确用法：
> "帮我把 src/utils/auth.ts 里的 token 验证逻辑拆出来，封装成一个独立的 validateToken 函数，错误处理保持不变，测试文件同步更新。"

错误用法：
> "JWT 验证怎么做？"

## 分层 Prompt 工作流

### Level 1：上下文注入（每次必做）

```bash
# 项目根目录放 CLAUDE.md
# 内容模板：
# 1. 项目技术栈和架构
# 2. 编码规范（命名、提交规范）
# 3. 目录结构说明
# 4. 当前项目阶段和问题
```

在项目根目录放一个 `CLAUDE.md` 或 `.cursorrules`，让 AI 每次都理解你的上下文。

### Level 2：任务分级

| 级别 | 描述 | 例子 |
|------|------|------|
| L1 | 单文件修改 | "修复这个类型错误" |
| L2 | 多文件联动 | "把 auth 模块迁移到新架构" |
| L3 | 端到端功能 | "实现完整的评论系统，含接口+前端+测试" |
| L4 | 架构设计 | "帮我设计这套微服务的通信方案" |

根据级别选择不同工具：
- L1/L2 → 直接用 AI 工具处理
- L3 → 需要先写 spec，再让 AI 实现
- L4 → AI 只能辅助思考，最终判断必须是人

### Level 3：Review & Commit

每次 AI 完成代码后，必须人工 review 两个东西：
1. **逻辑正确性**——AI 写的代码看起来对，但业务逻辑它不一定理解对了
2. **安全边界**——AI 对 SQL 注入、XSS 等安全问题有时会疏忽

## 实用场景清单

### 场景 1：代码审查

```bash
# 把 git diff 给 AI 审查
git diff HEAD~1 | claude --review "代码质量、安全、性能"
```

### 场景 2：Bug 定位

```bash
claude --task "分析 logs/app.log，找出 10:30 到 10:35 之间所有 500 错误的根因，给出修复建议"
```

### 场景 3：测试生成

```bash
claude --task "为 src/api/users.ts 的每个 endpoint 写集成测试，用 Jest，覆盖所有 error path"
```

### 场景 4：技术债务清理

```bash
claude --task "找出 src/ 下所有超过 200 行的函数，以及所有未使用的 export，列出优先级建议"
```

## 工具选型建议

| 工具 | 适合场景 |
|------|---------|
| Claude Code | 命令行重度用户，Terminal 原生 |
| Cursor | 需要实时协作、边写边看 |
| GitHub Copilot | 个人项目、轻量辅助 |
| Copilot Workspace | 探索性任务、头脑风暴 |

## 最重要的一个建议

**不要让它替代你思考，只让它替代你打字。**

写清楚你想要什么，review 它写出来的代码，这是人该做的事情。

AI 是执行层的放大器，不是判断层的替代品。
