# 为什么我放弃 GraphQL 回到 REST

GraphQL 刚出来那会儿，社区一片欢呼，仿佛 REST 的时代要结束了。我也在几个项目里实践了，最近决定全部回退到 REST。说说原因。

## 1. N+1 问题才是真正的噩梦

GraphQL 最大的坑就是 N+1。你以为一个 query 搞定，实际上一个列表页可能触发几十次数据库查询。

```python
# 这段 GraphQL Resolver
def resolve_posts(root, info):
    posts = db.query("SELECT * FROM posts LIMIT 20")
    for post in posts:
        post.author = db.query(f"SELECT * FROM users WHERE id={post.user_id}")  # N+1!
    return posts
```

DataLoader 确实能缓解，但需要全链路改造，老项目根本玩不动。

## 2. 缓存体系重建成本

HTTP 层缓存是 REST 自带的免费午餐。CDN、浏览器、网关，随便配。而 GraphQL 的缓存需要你自己实现。

> 用 REST 做 C端接口，CDN 一配，毫秒级响应。用 GraphQL，缓存自己写，bug 自己背。

## 3. 监控和调试成本

REST 有成熟的可观测性体系：请求日志、错误率、耗时，一条链路清清楚楚。GraphQL 的请求都长一个样，调试只能靠 Apollo Studio，付费的。

## 4. 什么场景确实适合 GraphQL

- BFF 层，后端聚合多个服务给前端用
- 多端 App 需要不同字段，版本管理简单
- 团队前后端完全分离，Schema 即契约

## 结论

**能用 REST 解决的问题，别用 GraphQL。** 只有在多端数据聚合场景下，GraphQL 的价值才能体现。其余时候，你只是在解决一个本来不存在的问题。

技术选型没有银弹，适合的才是最好的。
